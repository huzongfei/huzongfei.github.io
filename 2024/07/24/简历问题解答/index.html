IoC和AoP的原理

:blue_heart: IoC全程是inversion of control 控制反转，是一种设计原则，核心思想是将对象的创建和依赖的管理交由外部容器或框架处理，从而实现组件解耦和代码的灵活性。

主要通过依赖注入实现，依赖注入主要有构造函数注入，Setter方法注入，属性注入

:blue_heart: AOP（Aspect-Oriented Programming，面向切面编程）能够将那些与业务无关，却为业务模块所共同调用的逻辑或责任（例如事务处理、日志管理、权限控制等）封装起来，便于减少系统的重复代码，降低模块间的耦合度，并有利于未来的可扩展性和可维护性。

Spring AOP是基于动态代理的，如果要代理的对象实现了某个接口，那么SpringAOP就会使用JDK动态代理区创建代理对象，而对于没有实现接口的对象，无法使用JDK动态代理，转而使用CGlib动态代理生成一个被代理对象的子类作为代理对象。

主要应用：日志记录，安全检查、事务管理、异常处理

Mysql优化

1.读写分离：

将对数据库的读写操作分散到不同的数据库节点上。可以小幅度提高写性能，大幅度提高读性能

一主多从，主写从读，主从复制

常用方式

  • 代理方式 MySQL Router Atlas MaxScale MyCat
  • 组件方式 sharding-jdbc

主从复制原理：MySQL binlog(binary log 即二进制日志文件) 主要记录了 MySQL 数据库中数据的所有变化(数据库执行的所有 DDL 和 DML 语句)。因此，我们根据主库的 MySQL binlog 日志就能够将主库的数据同步到从库中。

主从同步步骤：

​ 主库将数据库中数据的变化写入到 binlog

​ 从库连接主库

​ 从库会创建一个 I/O 线程向主库请求更新的 binlog

​ 主库会创建一个 binlog dump 线程来发送 binlog ，从库中的 I/O 线程负责接收

​ 从库的 I/O 线程将接收的 binlog 写入到 relay log 中。

​ 从库的 SQL 线程读取 relay log 同步数据到本地（也就是再执行一遍 SQL ）

2.分库分表

分库 就是将数据库中的数据分散到不同的数据库上，可以垂直分库，也可以水平分库。

  • 垂直分库 就是把单一数据库按照业务进行划分，不同的业务使用不同的数据库，进而将一个数据库的压力分担到多个数据库。
  • 水平分库 是把同一个表按一定规则拆分到不同的数据库中，每个库可以位于不同的服务器上，这样就实现了水平扩展，解决了单表的存储和性能瓶颈的问题。

分表 就是对单表的数据进行拆分，可以是垂直拆分，也可以是水平拆分。

  • 垂直分表 是对数据表列的拆分，把一张列比较多的表拆分为多张表。
  • 水平分表 是对数据表行的拆分，把一张行比较多的表拆分为多张表，可以解决单一表数据量过大的问题。

什么情况下需要分库分表？

  • 单表的数据达到千万级别以上，数据库读写速度比较缓慢。
  • 数据库中的数据占用的空间越来越大，备份时间越来越长。
  • 应用的并发量太大（应该优先考虑其他性能优化方法，而非分库分表）。

分库分表带来的问题

join 操作：同一个数据库中的表分布在了不同的数据库中，导致无法使用 join 操作。这样就导致我们需要手动进行数据的封装，比如你在一个数据库中查询到一个数据之后，再根据这个数据去另外一个数据库中找对应的数据。

事务问题：同一个数据库中的表分布在了不同的数据库中，如果单个操作涉及到多个数据库，那么数据库自带的事务就无法满足我们的要求了。这个时候，我们就需要引入分布式事务了。

分布式 ID：分库之后， 数据遍布在不同服务器上的数据库，数据库的自增主键已经没办法满足生成的主键唯一了。我们如何为不同的数据节点生成全局唯一主键呢？这个时候，我们就需要为我们的系统引入分布式 ID 了。

跨库聚合查询问题：分库分表会导致常规聚合查询操作，如 group by，order by 等变得异常复杂。这是因为这些操作需要在多个分片上进行数据汇总和排序，而不是在单个数据库上进行。为了实现这些操作，需要编写复杂的业务代码，或者使用中间件来协调分片间的通信和数据传输。

3.数据冷热分离

数据冷热分离是指根据数据的访问频率和业务重要性，将数据分为冷数据和热数据，冷数据一般存储在存储在低成本、低性能的介质中，热数据高性能存储介质中。

如何区分冷热数据：

​ 时间维度区分

​ 访问频率区分

数据冷热分离的优缺点

优点：热数据的查询性能得到优化（用户的绝大部分操作体验会更好）、节约成本（可以冷热数据的不同存储需求，选择对应的数据库类型和硬件配置，比如将热数据放在 SSD 上，将冷数据放在 HDD 上）

缺点：系统复杂性和风险增加（需要分离冷热数据，数据错误的风险增加）、统计效率低（统计的时候可能需要用到冷库的数据）。

4.SQL优化手段(常见)

  • 避免使用select*
  • 分页优化： 普通分页查询—>子查询分页（会产生一张新表，会影响性能）
  • 避免多表做join
  • 建议不要使用外键与级联
  • 选择适合的字段类型
  • 尽量用UNION ALL 代替 UNION
  • 批量操作

Docker常见命令

Redis

主从复制 ： 单节点的Redis并发能力有限，通过搭建主从集群提高Redis的并发能力，实现读写分离。一般都是一主多从，主节点负责写，从节点负责读。主从同步分为两种，全量同步和增量同步。 replication id 和 offset.

哨兵模式：保证Redis的高并发，高可用

Redis哨兵机制是通过在独立的哨兵结点上运行特定的哨兵进程来实现的。这些哨兵进程监控主从节点的状态，并在发现故障时自动完成故障发现和转移，并通知应用方，实现高可用性。

哨兵选举：在启动时，每个哨兵节点会执行选举过程，其中一个哨兵节点被选为leader,负责协调其他哨兵节点。

  • 每个在线的哨兵节点都可以成为领导者，每个哨兵节点回向其他哨兵发**is-master-down-by-addr **命令，征求判断并要求将自己设置为领导者
  • 当其他哨兵收到此条消息时，可以同哟或者拒绝它成为leader
  • 如果哨兵发现自己在选举的票数大于等于num(sentinels)/2+1时，成为leader,如果没有超过，继续选举。

缓存穿透：查询一个一定不存在的数据，每次查询都会到数据库中进行查询，就可能导致数据库挂掉。解决方案：布隆过滤器，缓存空对象

什么是布隆过滤器。布隆过滤器主要是用于检索一个元素是否在一个集合中。初始化一个较大的数组，对一个key经过多次hash计算进行查询。

缓存击穿 ：对于设置了过期时间的key，缓存在某个时间点过期的时候，恰好这个点对这个key有大量的并发请求过来，这些请求发现缓存过期后从数据库加载数据并设置缓存，大并发的请求可能会瞬间把数据库压垮。解决方案：使用互斥锁，设置key逻辑过期

缓存雪崩：设置缓存时采用了相同的过期时间，导致缓存在某一时刻同时失效，请求全部转发到DB，DB 瞬时压力过重雪崩。与缓存击穿的区别：雪崩是很多key，击穿是某一个key缓存。解决方案：在过期时间上加随机值

:heartpulse: 数据库和缓存一致性问题

不管是先写MySQL数据库，再删除Redis缓存；还是先删除缓存，再写入MySQL数据库，都有可能出现数据不一致的情况。

解决方案：

延迟双删 — 先删除Redis缓存数据，再更新MySQL,延迟几百毫秒再删除Redis缓存数据，这样就算再更新MySQL时，有其他线程读了MySQL,把老数据读到了Redis,那么也会被删除掉，从而把数据保持一致。

队列+重试机制 — 更新数据库数据；缓存因为种种问题删除失败；将需要删除的key发送至消息队列； 自己消费消息，获取需要删除的key； 继续重试删除操作，直到成功

异步更新缓存(基于订阅binlog的同步机制) — MySQL中产生了新的写入、更新、删除等操作，就可以把binlog相关的消息推送到Redis,Redis再根据binlog中的记录，对Redis进行更新。

消息队列RabbitMQ

https://www.cnblogs.com/davidgu/p/14702449.html

核心组件:

  • Broker :RabbitMQ服务器，负责接收和分发消息的应用

  • Virtual Host: 虚拟主机，是RabbitMQ中的逻辑容器，用于隔离不同环境或不同应用程序的信息流。每个虚拟主机都有自己的队列、交换机等设置，可以理解为一个独立的RabbitMQ服务。

  • Connection连接：虚拟主机，是RabbitMQ中的逻辑容器，用于隔离不同环境或不同应用程序的信息流。每个虚拟主机都有自己的队列、交换机等设置，可以理解为一个独立的RabbitMQ服务。

  • Channel通道：是在Connection 内创建的轻量级通信通道，用于进行消息的传输和交互。应用程序通过Channel进行消息的发送和接收。通常一个Connection 可以建立多个 Channel。

  • Exchange交换机：交换机是消息的中转站，负责接收来自生产者的消息，并将其路由到一个或多个队列中。RabbitMQ 提供了多种不同类型的交换机，每种类型的交换机都有不同的消息路由规则。

    交换机类型

    • direct(直连交换机)：具有路由功能的交换机，绑定到此交换机的时候需要指定一个routing_key，交换机发送消息的时候需要routing_key，会将消息发送道对应的队列。
    • topic(主题交换机)：在直连交换机基础上增加模式匹配，也就是对routing_key进行模式匹配，*代表一个单词，#代表多个单词。
    • headers(头部交换机)：忽略routing_key，使用Headers信息(一个Hash的数据结构)进行匹配，优势在于可以有更多更灵活的匹配规则。
    • fanout(扇形交换机)：广播消息到所有队列，没有任何处理，速度最快。

  • Queue队列：队列是消息的存储位置。每个队列都有一个唯一的名称。消息从交换机路由到队列，然后等待消费者来获取和处理。

  • Binding绑定关系： Binding 是 Exchange 和 Queue 之间的关联规则，定义了消息如何从交换机路由到特定的队列。

集群构建：普通集群（默认）和镜像集群

消息队列模式：

workQueue 工作序列机制： Producer 将消息发送到 queue，多个 Consumer 同时消费Queue 上的消息。消息会均匀的分配给多个 Consumer 处理。

Publish/Subscribe 订阅发布机制： Producer 只负责将消息发送到exchange交换机上。Exchange 将消息转发到所有订阅的 Queue，并由对应的 Consumer 去进行消费

Routing 基于内容路由机制：在订阅发布机制的基础上，增加一个routingKey，并根据routingKey判断 Exchange 将消息转发到哪些 Queue 上。

Topic 基于话题路由机制：在基于内容路由的基础上，对routingKey增加了模糊匹配的功能。

死信队列：死信队列（DLQ）是一种特殊类型的消息队列，用于临时存储软件系统由于错误而无法处理的消息。

当一条消息在队列中出现以下三种情况的时候，该消息就会变成一条死信。

  • 消息被拒绝(basic.reject / basic.nack)，并且requeue = false
  • 消息TTL过期
  • 队列达到最大长度

当消息在一个队列中变成一个死信之后，如果配置了死信队列，它将被重新publish到死信交换机，死信交换机将死信投递到一个队列上，这个队列就是死信队列。

Spring Cloud Gateway

:blue_heart:网关功能：统一接入、流量管控、协议适配、安全维护

:blue_heart:三大核心概念：路由（Route）、断言（Predicate）、过滤器（Filter）

:blue_heart: 如何进行路由匹配：Spring Cloud Gateway 通过定义路由规则来进行路由匹配。每个路由包括一个ID，一个目标URL，一组断言(Predicates)和一组过滤器(Filters)。路由规则通过断言来匹配请求。

​ 断言— 一个java函数，用于根据请求的某些属性(如路径，头信息，参数等)决定是否将请求路由到特定的目标。

​ 过滤器— 用于在请求被路由前后者响应返回客户端前对请求或响应进行修改。内置过滤器：AddRequestHeader、AddResponseHeader、RewritePath等

:blue_heart:如何实现限流：Spring Cloud Gateway 通过RateLimiter过滤器来实现限流。内置的限流器包括RedisRateLimiter和RequestRateLimiter使用RedisRateLimiter需要配置Redis。

:blue_heart: 如何实现重试：Spring Cloud Gateway通过Retry过滤器来实现重试机制。Retry过滤器允许在请求失败时进行重试，可以配置重试的最大次数、重试的条件等。

:blue_heart: 如何在Spring Cloud Gateway中自定义过滤器：自定义过滤器需要实现GatewayFilter或者GlobalFilter接口.GatewayFilter是作用于特定路由的过滤器，GlobalFilter是全局过滤器，作用与所有路由

分布式事务

Seata —-一款开源的分布式事务解决方案，致力于在微服务架构下提供高性能和简单易用的分布式事务服务

CAP原理 ：一致性（Consistency）可用性（Availability） 分区容错性（Partition tolerance）

BASE理论:BASE是CAP理论中AP方案的延伸，核心思想是即使无法做到强一致性（StrongConsistency，CAP的一致性就是强一致性），但应用可以采用适合的方式达到最终一致性（Eventual Consitency）。

  • **Basically Available(基本可用)**：基本可用是指分布式系统在出现不可预知的故障的时候，允许损失部分可用性，但不等于系统不可用。

  • Soft state(软状态)：允许系统中的数据存在中间状态，并认为该中间状态的存在不会影响系统的整体可用性，即允许系统在不同节点的数据副本之间进行数据同步的过程存在延时。

  • **Eventually consistent(最终一致性)**：强调系统中所有的数据副本，在经过一段时间的同步后，最终能够达到一个一致的状态。

AP思想：各个子事务分别执行和提交，无需锁定数据。允许出现结果不一致，然后采用弥补措施恢复，实现最终一致性。如AT模式

CP思想：各个子事务执行后不要提交，而是等待彼此结果，然后同时提交或回滚。在这个过程中锁定资源，不允许其他人访问，数据处于不可用状态，但能保证一致性。如XA模式

Seata支持四种不同的分布式解决方案：XA,TCC,AT,SAGA

XA模式：两阶段提交 —– 同时提交

一阶段：

  • 事务协调者通知每个事务参与者执行本地事务
  • 本地事务执行完成后报告事务执行状态给事务协调者，此时事务不提交，继续持有数据库锁

二阶段：

  • 事务协调者基于一阶段的报告来判断下一步操作
  • 如果一阶段都成功，则通知所有事务参与者，提交事务
  • 如果一阶段任意一个参与者失败，则通知所有事务参与者回滚事务

AT模式：—– 分别提交，异常回滚

第一阶段：记录数据快照，执行并提交事务

第二阶段：根据阶段一的结果判断

​ 如果每一个分支事务都成功，则事务已经结束（一阶段已经提交），删除阶段一的快照

​ 如果有任意分支事务失败，则需要根据快照恢复到更新前数据，然后删除快照

可能出现脏写问题，解决思路 引入全局锁

TCC模式：Try-Confirm-Cancel

阶段一（ Try ）：检查余额是否充足，如果充足则冻结金额增加30元，可用余额扣除30

**阶段二（Confirm)**：假如要提交（Confirm），之前可用金额已经扣减，并转移到冻结金额。因此可用金额不变，直接冻结金额扣减30即可

**阶段二(Cancel)**：如果要回滚（Cancel），则释放之前冻结的金额，也就是冻结金额扣减30，可用余额增加30

瑞集商城

项目介绍：瑞集商城项目是基于单体项目过于冗余的背景，通过Spring Cloud Alibaba组件对单体应用进行微服务化改造，将该项目划分为商品微服务，购物车微服务，用户微服务，支付微服务。首先是对数据库进行划分。然后利用Nacos实现服务注册，OpenFeign实现远程调用。

人员架构

担任角色

项目主要负责模块

通过Spring Coud Alibaba 对单体应用进行微服务化改造，将项目划分为商品微服务，购物车微服务，用户微服务，交易微服务，支付微服务；

怎么划分的，说一下步骤：

  1. 建立不同的数据库，每个微服务对应不同的微服务，如商品微服务对应product-service 数据库
  2. 从单体项目中将服务抽取出来，以商品服务为例，首先建立商品微服务项目，然后引入依赖，编写启动类，编写配置文件（可以从原本的单体项目中按需拷贝，并做出一定修改，数据库改变）

利用Nacos实现服务注册，然后利用OpenFeign远程服务调用，实现各个微服务之间的相互调用，如在交易微服务中调用商品微服务根据id批量查询商品功能；

什么是OpenFeign：OpenFeign是一个声明式的http客户端，是SpringCloud 在Eureka公司开源的Feign基础上改造来的。

服务雪崩:服务雪崩是指在分布式系统中，由于一个或多个服务的故障，导致依赖这些服务的其他服务也发生故障，最终引起整个系统的崩溃。——需要避免

防止雪崩：

  1. 服务限流：服务限流（Rate Limiting）是一种通过限制系统接收和处理请求的速率来保护系统的措施。限流的主要目的是防止系统过载，从而确保系统的稳定性和可靠性。

  2. 线程隔离：限流可以降低服务器压力，尽量减少因并发流量引起的服务故障的概率，但并不能完全避免服务故障。一旦某个服务出现故障，我们必须隔离对这个服务的调用，避免发生雪崩。

  3. 服务降级：服务降级（Service Degradation）是一种在服务不可用或性能下降时，通过提供替代方案来减少对用户影响的措施。服务降级的主要目的是保证核心功能的可用性，避免全局性故障。

  4. 服务熔断:服务熔断是一种保护措施，用于防止一个服务的故障扩散到整个系统。当某个服务的故障达到一定阈值时，熔断器会触发，暂时停止对该服务的调用，以便让该服务有时间恢复。

    • 合并 CompletableFuture： 将所有的 CompletableFuture 添加到一个列表中，然后使用 CompletableFuture.allOf 方法等待所有任务完成。CompletableFuture.anyof()最快完成的任务返回
    • 获取结果并合并： 当所有任务完成后，使用 thenApply 方法将所有的 CompletableFuture 的结果合并为一个列表。

    • 关闭线程池： 关闭所有使用的线程池。

如何创建线程池

线程池的三大方法，七大参数，四种拒绝策略

线程池的好处: 降低资源的消耗 提高响应的速度 方便管理

阿里规范中提到，创建线程池必须使用ThreadPoolExexctor，指定其七个参数，分别是：

  1. corePoolSize: 核心线程数

  2. maximumPoolSize:最大线程数

  3. keepAliveTime:空闲线程存活时间（超过这个时间会被回收） —–针对非核心线程

  4. unit:时间单位 keepAliveTime的时间单位

  5. workQueue:工作队列 存放待执行任务的队列：当提交的任务数超过核心线程数大小后，再提交的任务就存放在工作队列，任务调度时再从队列中取出任务。它仅仅用来存放被execute()方法提交的Runnable任务。工作队列实现了BlockingQueue接口

    JDK默认的工作队列有五种：

    • ArrayBlockingQueue 数组型阻塞队列; 数组结构，初始化时传入大小，有界，FIFO，使用一个重入锁，默认使用非公平锁，入队和出队共用一个锁，互斥。

    • LinkedBlockingQueue 链表型阻塞队列：链表结构，默认初始化大小为Integer.MAX_VALUE，有界（近似无界），FIFO，使用两个重入锁分别控制元素的入队和出队，用Condition进行线程间的唤醒和等待。

    • SynchronousQueue 同步队列：容量为0，添加任务必须等待取出任务，这个队列相当于通道，不存储元素。

    • PriorityBlockingQueue 优先阻塞队列：无界，默认采用元素自然顺序升序排列。

    • DelayQueue 延时队列：无界，元素有过期时间，过期的元素才能被取出。

  6. threadFactory：线程工厂; 创建线程的工厂，可以设定线程名、线程编号等。

  7. handler: 拒绝策略。当线程池线程数已满，并且工作队列达到限制，新提交的任务使用拒绝策略处理。可以自定义拒绝策略，拒绝策略需要实现RejectedExecutionHandler接口。

    JDK默认的拒绝策略有四种：

    • AbortPolicy：丢弃任务并抛出RejectedExecutionException异常。
    • DiscardPolicy：丢弃任务，但是不抛出异常。可能导致无法发现系统的异常状态。
    • DiscardOldestPolicy：丢弃队列最前面的任务，然后重新提交被拒绝的任务。
    • CallerRunsPolicy：由调用线程处理该任务。

什么是微服务，使用微服务有什么好处，有什么缺点

微服务是一种软件架构设计模式，他将单一应用程序拆分成一组小型服务，每个服务都运行在自己的进程中，并通过一定的通信机制进行通信，主要优缺点有：

优点：

  1. 每个微服务可以独立部署和扩展，根据业务需求灵活地扩展或者缩减特定服务的资源
  2. 每个微服务可以使用不同的技术栈，开发团队可以根据需求选择
  3. 服务之间松耦合，一个服务的故障不会影响整个系统的运行（服务降级和服务熔断）
  4. 微服务的小型，独立特性使得开发团队可以快速迭代和部署新功能

缺点：

  1. 微服务架构引入了分布式系统的复杂性，需要处理服务发现、负载均衡、容错等问题
  2. 服务数量增多，运维和监控的复杂度也相对应增加，运维成本增加
  3. 分布式系统中，实现业务数据的一致性变得更加复杂
  4. 各个微服务之间相互依赖，端到端的测试和调试更加困难
  5. 增加网络开销

各个微服务之间怎么调用

服务之间的相互调用通常可以归类为两种主要模式：**RPC（Remote Procedure Call）和消息模式**。

  1. RPC（Remote Procedure Call）:RPC是一种同步调用模式，它使得服务之间的通信看起来像是在调用本地方法。RPC 可以使用多种协议和技术实现，包括 HTTP、gRPC、Thrift 等。以下是一些关键点：

    1. 同步通信：调用者会等待被调用的服务返回结果。这使得 RPC 模式适用于需要立即获得响应的场景。

    2. 接口定义：通常使用接口定义语言（IDL）来定义服务接口，例如 gRPC 使用 Protocol Buffers。

    3. 协议支持：RPC 可以基于多种传输协议，如 HTTP、HTTP/2、TCP 等。

    4. 实现方式：

      • HTTP/RESTful API：通过 HTTP 协议和 RESTful 风格的 API 进行通信。工具如 Spring RestTemplate、Spring WebClient、OpenFeign。

      • gRPC：基于 HTTP/2 的高性能 RPC 框架，使用 Protocol Buffers 作为 IDL。

      • Thrift：由 Apache 开发的跨语言 RPC 框架。

  2. 消息模式是一种异步通信模式，通过消息队列实现服务之间的通信。消息模式适用于解耦、异步处理和提高系统弹性的场景。以下是一些关键点：

    1. 异步通信：消息发送者将消息发送到消息队列或主题，接收者异步地从队列中消费消息。
    2. 松耦合：消息发送者和接收者之间没有直接的依赖关系，增加了系统的灵活性和可扩展性。
    3. 消息中间件：使用消息中间件（如 RabbitMQ、Apache Kafka、ActiveMQ）来管理消息的发布和消费。
    4. 实现方式：
      • 消息队列：发送者将消息发送到队列，接收者从队列中消费消息，实现异步处理。
      • 发布-订阅（Pub/Sub）：消息发布者发布消息到主题，多个订阅者可以订阅并处理这些消息。

什么是反向代理：

反向代理是一种代理服务器，它接收客户端的请求，然后将请求转发到后端服务器，并将后端服务器的响应返回给客户端。反向代理与传统的正向代理不同，正向代理主要用于代理客户端的请求以访问互联网资源，而反向代理主要用于代理服务器端的资源提供给客户端。

优点：

反向代理服务器可以隐藏后端服务器的 IP 地址和其他信息，从而提高安全性。

反向代理服务器可以将请求分发到多个后端服务器，均衡负载，提高系统的可靠性和性能。
